# v0.4 — per-track camera altitude

Goal: the camera picks an appropriate altitude for each track automatically.
One altitude per render (held constant for the whole flyover); different
tracks get different values based on their own geometry.

## the problem, grounded in evidence

Two renders, same camera strategy, very different outcomes:

- **ACT (Annapurna demo, 45 km · 2907–5416 m):** trail reads as a clear
  orange line threading a dramatic valley. Snow peaks on either side give
  depth. The current fixed altitude happens to work.
- **亚莫措根 (Kham plateau, shorter track):** trail collapses to a single
  red dot lost in a sea of mid-tone mountains. You can barely tell a
  journey is happening. Same camera, very different geometry.

Root cause: camera altitude is a hard-coded constant in
`public/camera/smooth-constant.js`. It was implicitly tuned for Annapurna
during development. Any track whose local terrain relief, trail length,
or surrounding-peak density doesn't match that regime looks wrong.

## scope

**In:**

- Per-track altitude heuristic derived from GPX-computable inputs:
  - track total distance,
  - elevation range (max - min) after DEM calibration,
  - bbox diagonal,
  - optionally: nearest-peak prominence within N km of the trail (we already
    fetch POIs in `queryRawPOIs`).
- Explicit override: `--camera-altitude METERS` (CLI) + advanced UI field.
  Same pattern as `--pace` / `--intro`.
- Instrumentation: chosen altitude surfaced in the `plan` event so the UI
  log shows it; add a unit test pinning the formula.
- Calibration: render the corpus (ACT + 亚莫措根 + one flat coastal +
  one short city-park GPX) with the new heuristic and eyeball it.

**Out (deferred to v0.5+):**

- Intra-track altitude variation ("camera follows terrain"). User's
  explicit call.
- Pitch that adapts to slope.
- Dwell-stop orbit.
- Zoom variation by local terrain density.

Keeping it narrow. v0.4 changes ONE camera parameter as a function of
ONE derived scalar. If that single change is measurably better across
the corpus, the formula is right; if not, we iterate.

      `trackData` (has bounds, totalDistance, points, pois). Output: a
      scalar meters value. Ship a first-pass formula; tune on corpus.
- [ ] **Pass `cameraAltitude` through the pipeline** — `plan` event,

- [ ] **`--camera-altitude METERS` CLI flag + advanced UI field.** For

## working notes / hypotheses

First-pass formula candidate (not yet verified):

```
cameraAltitude = clamp(
  meanElevation + (maxEle - minEle) * K,
  FLOOR,    // e.g. 400 m — keep camera off the trees
  CEILING,  // e.g. 3500 m — avoid cartoon-high drone shot on huge tracks
)
```

For ACT: mean ~4100 m, range 2500 m, K=0.6 → camera absolute altitude
5600 m. That's near the current implicit value, so the formula is
coherent with the good case. For a track with 500 m elevation range on
a plateau, same K yields mean + 300 m, which pulls the camera in much
closer — exactly what 亚莫措根 needs.

K is the tuning knob. Calibration corpus picks the value.

If the formula fails on the corpus — e.g. there's a low-relief but very
long coastal track where "mean + K*range" is still too close — add
`totalDistance` as a second input. Keep it to scalars, resist per-point
adaptation (that's v0.5).
