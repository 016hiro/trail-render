# v0.3 — production polish ✅

Goal: the web UI ships and survives an unattended weekend of strangers
using it. No new features; close the gaps that make the existing surface
fragile.

Status: **all must-have + nice-to-have items shipped 2026-04-15.** ETA
display for queued jobs is the only deferred item (rolled into v0.4).

## must-have

- [x] **`output/jobs/` TTL.** Server purges directories older than
      `JOB_TTL_DAYS` (default 30) at startup. Configurable via env;
      set to 0 to disable.
- [x] **Cancel a running render.** `DELETE /api/jobs/:id` returns
      `{status: 'cancelling'}` for running, `{status: 'cancelled'}` for
      queued; UI grew a `▣ CANCEL RENDER` button. AbortSignal flows
      through job-runner → captureFrames (which closes the headless
      browser) and ffmpeg (which is now spawned, not execSync'd, so
      we can SIGTERM it).
- [x] **Concurrent renders → real queue.** FIFO `queue[]` array,
      `runningJobId` slot. Second submit returns
      `{status: 'queued', queuePosition: N}`. `tryStartNext()` runs
      after every settle. Cancellation of a queued job removes from
      queue without ever starting it. (Queue is in-memory only — disk
      persistence deferred; if restart drops pending jobs, that's a
      v0.4 wishlist item once we feel the pain.)
- [x] **Resume in-progress render after refresh.** Submit handler does
      `history.replaceState(null, '', '?job='+jobId)`. The `?job=` IIFE
      on page load now routes by status: queued/running → subscribe;
      done → onDone; cancelled → cancelled state; error → showError;
      no in-memory record → disk-fallback download URL.
- [x] **Disable form fields during render.** Wrapped the form contents
      in `<fieldset id="options-fieldset">`; toggled `disabled` on
      submit and re-enabled on terminal events. CSS dims the panel
      and the active toggle button so it reads as inactive.

## nice-to-have

- [x] **SSE auto-reconnect with backoff.** Subscribe loop now wraps
      EventSource in a `connect()` closure: onerror reschedules after
      backoff (1 s → 2 s → … capped at 30 s), `terminal=true` short-
      circuits the retry on done/cancelled/error. UI shows a blinking
      "reconnecting …" indicator next to the cancel button.
- [x] **Per-job log file persisted next to the MP4.** Each job opens
      an append-only `output/jobs/<id>/log.jsonl` (one JSON event per
      line) when enqueued; closed on terminal event. Survives server
      restart; useful for post-mortems.
- [x] **`/api/status` exposes queue depth.** Returns
      `{busy, busyJobId, queue: [ids], queueDepth, jobCount}`.
      `/api/jobs/:id` exposes `queuePosition` for queued items.
      ETA-for-queued is deferred — we don't have a per-job duration
      estimate yet; promote to v0.4 once it's worth showing.

## bug-watch (carry-over from devlog 2026-04-15)

- [x] **SSE-close fix verified end-to-end.** Real render through web UI
      reached `done` cleanly, video played, no replay-induced reset
      observed. The defensive `lastDoneJobId` idempotency guard never
      had to fire — primary `es.close()` on terminal events handled it.
- [x] **Audit DOM self-assignment hazards.** Swept `public/` for `.src`,
      `.href`, `setAttribute('src'…)`, and `createElement('img'|'iframe'
      |'video'|'audio'|'source'|'script'|'link'|'embed'|'object')`.
      Only two hits: `#done-video.src = url` (already guarded by
      `lastDoneJobId` — bug 3 fix) and `#download-btn.href = url`
      (anchors don't reload on self-assignment, safe). No dynamic
      resource-element creation anywhere. Nothing to fix.
