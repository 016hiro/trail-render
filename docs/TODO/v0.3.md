# v0.3 — production polish

Goal: the web UI ships and survives an unattended weekend of strangers
using it. No new features; close the gaps that make the existing surface
fragile.

## must-have

- [ ] **`output/jobs/` TTL or cap.** Today every render leaves an `input.gpx`
      + `trail.mp4` on disk forever. A 30-day-old MP4 cap, or a per-disk
      total-bytes cap, prevents the host from filling up.
- [ ] **Cancel a running render.** Once a user clicks COMMIT they have no
      way out short of killing the server. Add `DELETE /api/jobs/:id` →
      kills the playwright + ffmpeg subprocesses; UI grows a "stop" button
      in the running state.
- [ ] **Concurrent renders → real queue.** Currently the second submit
      gets 409. Acceptable for one user, hostile when shared. FIFO
      queue, persisted to disk so a server restart doesn't drop pending
      jobs. Cap concurrency at 1 (chromium + ffmpeg are heavy).
- [ ] **Resume in-progress render after refresh.** The jobId already
      survives in `output/jobs/<id>/`; surface it in the URL during the
      run (`/?job=<id>`) and let `?job=` *always* try SSE first. Today
      a refresh during a render strands the UI.
- [ ] **Disable form fields, not just the submit button, during render.**
      Right now the user can edit `--pace` mid-render with no effect —
      misleading. Disable the whole form via `<fieldset disabled>`.

## nice-to-have

- [ ] **SSE auto-reconnect with backoff for genuine network blips.**
      v0.2's fix correctly closes ES on `done`/`error`, but a mid-render
      disconnect (laptop sleep, wifi blip) currently just stops updating
      the UI silently. Detect onerror while running, show a "reconnecting…"
      indicator, retry with exponential backoff up to ~30s.
- [ ] **Per-job log file persisted next to the MP4.** `output/jobs/<id>/log.txt`
      with the same lines the CLI emits. Helps post-mortems for "why did
      this render look weird" questions; also lets `?job=<id>` show the
      log even after restart.
- [ ] **Server `/api/status` exposes queue depth + busy job ETA.** UI can
      show "1 ahead of you, ~6 min wait" instead of a blunt 409.

## bug-watch (carry-over from devlog)

- [ ] Verify the SSE-close fix holds end-to-end on a real render. It was
      validated by inspection + reverse test in the dev-tools console;

- [ ] Audit other places where assigning a DOM property to its current
